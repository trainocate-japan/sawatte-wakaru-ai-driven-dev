# Cursorルール設定例（そのまま貼れる雛形）

このファイルは、演習04で使う **Cursor のコーディングルール例** です。  
チームで少し調整したうえで、そのまま Cursor の Rules / Project Instructions に貼り付けて使えます。

## 使い方
1. まずは下の「貼り付け用テンプレート」をそのまま Cursor に入れる
2. 使用技術やチーム方針に合わせて、不要な行を削る
3. 実装前に Agent に「以下のルールに従って実装してください」と伝える
4. レビュー時にも同じルールを再提示して使う

## 貼り付け用テンプレート

```txt
このプロジェクトは、初学者向けの学習用プロジェクトです。
オンライン予約システムの最小実装を目的としています。
以下のルールを必ず守ってください。

# 目的
- まずはローカルで動く最小構成を作る
- 要件・仕様・設計にある内容を優先して実装する
- 学習用として読みやすいコードにする

# コーディングルール
1. 変数名・関数名は、役割が分かる名前にする
   - 省略しすぎた名前は使わない
   - data や temp のような意味の弱い名前を多用しない

2. 1つの関数に役割を詰め込みすぎない
   - 1つの関数では1つのまとまった責務を中心に扱う
   - 長すぎる関数は分割案を出す

3. 初学者が読める書き方を優先する
   - 難しい構文や過度に高度な書き方は避ける
   - 短く書けても、読みにくくなる書き方は採用しない

4. ファイル数を増やしすぎない
   - 学習用の最小構成を優先する
   - 分割が必要な理由がある場合だけファイルを増やす

5. 入力チェックとエラー処理を入れる
   - 必須入力が不足している場合は分かりやすく知らせる
   - 利用者に伝わるエラーメッセージにする
   - 想定される失敗を無視しない

6. コメントは必要最小限にする
   - コメントで補う前に、名前や処理の分け方で意図が分かるようにする
   - ただし、初学者が誤解しやすい箇所には短い補足を入れてよい

7. 要件・仕様・設計とずれる実装を勝手に追加しない
   - 不明点は仮定として明示する
   - 新しい機能を足す前に確認事項として示す

8. 実装前に、作成するファイル一覧と役割を先に示す
   - いきなりコードを書き始めず、最初に構成案を出す
   - 各ファイルの役割を短く説明する

9. コードを提案するときは、ルールをどう反映したかを説明する
   - どのルールを特に意識したかを短く添える

10. レビューを依頼されたときは、次の観点を優先する
   - 命名の分かりやすさ
   - 関数の長さと責務の分離
   - 入力チェックとエラー処理
   - 初学者にとっての読みやすさ
   - 要件・仕様・設計とのズレ
```

## 最小版（時間がないとき用）

```txt
このプロジェクトは初学者向けの学習用実装です。以下のルールで実装してください。

- 変数名と関数名は意味が分かる名前にする
- 1つの関数に役割を詰め込みすぎない
- 初学者が読めるシンプルな書き方を優先する
- ファイル数を増やしすぎない
- 必須入力のチェックと分かりやすいエラー表示を入れる
- 要件・仕様・設計にない機能を勝手に追加しない
- 実装前にファイル一覧と役割を先に示す
- レビュー時は、命名・関数分割・入力チェック・読みやすさを優先して確認する
```

## チームで調整しやすい追加例

### HTML / CSS / JavaScript の場合
```txt
- DOM操作は分かりやすい関数に分ける
- 画面表示の処理とデータ操作の処理をできるだけ分ける
- localStorage を使う場合は、保存と読み込みの処理をまとめる
```

### Python の場合
```txt
- ルーティング、画面表示、データ処理の役割をできるだけ分ける
- まずは標準機能やシンプルな構成で実装する
- 外部ライブラリは必要最小限にする
```

### レビューを厳しめにしたい場合
```txt
- 同じような処理が重複していたら共通化案を出す
- 例外ケースを無視している処理があれば優先的に指摘する
- 修正提案では、優先度を「高・中・低」で示す
```
